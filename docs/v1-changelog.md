## Per-phase highlights

### Phase 3: Talent Scout port

Talent Scout was originally shipped as a standalone Lovable app and lifted into HQ as an admin-only sub-route. The port covered schema augmentation, roles CRUD + wizard, the chunked Gmail pull pipeline (`BATCH_SIZE=8`), CandidateDetail + status dropdown + re-eval, the Final Review packet (pdf-lib + signed-URL email), referral ingestion (forward-chain walker), pg_cron + 6 cron jobs (scheduled pulls + 3 watchdogs + storage cleanup + monthly spend reset), scorecard refinement, and substantive describer fields. Per-sub-phase decisions in `docs/decisions.md` Phase 3 sections.

### Phase 4: Venue Scout port

Full 1:1 port from `mirror-nyc-venue-scout-pro` shipped to production 2026-05-13. Producer flow: New Scout → Brief upload (DOCX + AI parse) → Sheet Prompt → Sheet Upload (XLSX/CSV + AI enrichment) OR direct AI venue research → Sourcing Report → Shortlist (matrix + photo upload) → Review → Compiling (per-venue overview generation) → Deck Prep → Generating (Google Slides deck) → deck opens in new tab. AI surface uses `pause_turn` continuation, `writeFailure` CAS guards on all 3 AI edge functions, 180-360s `WORK_TIMEOUT_MS`, URL-fallback helpers. Phase 4 Revision rebuilt the Brief as a 3-step stepper (Event → Venue → Review), added venue-side fields the AI sourcing prompt needs, added `vs-generate-brief-overview`, made the Revisit nav always-visible. Per-sub-phase decisions in `docs/decisions.md` § "Phase 4 cutover + port plan locked decisions" + § "Phase 4 Revision".

### Phase 5.1 to 5.6: HQ Core foundations to interaction primitives

The first six HQ Core phases established the data backbone (5.1 permission tiers + `notes_log` + AppShell; 5.2 entities + clients-vendors split + DataTable / BoardView / TimelineView / CalendarMonthView primitives; 5.3 Calendar + Outlook + `promote_outlook_to_project` RPC) and the cross-tool surfaces (5.4 Wiki + Settings + admin pre-provisioning; 5.5 Notifications + Activity Feed + Search + Sign-in; 5.6 cross-cutting interaction primitives + inline-edit + global default views via `users.is_owner` + `saved_views.scope`).

### Phase 5.15: Anthropic per-tool call-log infra + spend breakdown surface

4 sub-phases (originally 5.15 / 5.15.1 / 5.15.2 / 5.15.3) collapsed into one squash. Closed out the Phase 5.12.14.3 R7 § F.2 placeholder ("Per-tool breakdown coming when the call-log infrastructure ships"), consolidated cap controls onto HQ Admin Settings, broadened the breakdown surface, canonicalized the list-table chrome HQ-wide, and added a Month / Year window selector. Four concerns landed across the four sub-phases; per-concern narrative preserved as paragraphs below.

**Infra + cap-control consolidation (originally Phase 5.15).** Schema (`20260608000000_phase_5_15_anthropic_call_log.sql`): new append-only `public.anthropic_call_log` table (id, created_at, app, fn_name, model, token columns, cost_usd, nullable `scout_id` / `role_id` FKs with `ON DELETE SET NULL`). Four indexes: `(app, fn_name, created_at desc)` for breakdown rollups, `(created_at)` for prune scans, partial indexes on `scout_id` + `role_id` for future drilldown. RLS enabled with single admin-only `SELECT` policy; service-role writes bypass RLS so no INSERT policy needed. Original aggregation RPC `public.anthropic_spend_breakdown(month_iso text default null)` — superseded by the windowed form below. Admin gate inside the body via `public.is_admin()`; EXECUTE granted to `authenticated` so non-admin callers surface a clean `admin only` exception instead of a 403; `revoke execute ... from public` per the Phase 5.8.5 SECURITY DEFINER hardening posture.

Wrapper (`_shared/anthropic.ts`): `CallClaudeOptions` gains optional `scout_id?: string | null` + `role_id?: string | null`. New `logCallToTable` helper writes one row after `trackSpendAndAlert`; both are best-effort, errors swallowed, so a logging outage never blocks the caller. `findVenueWebsite` options grow `scout_id?: string | null` and pass it through to its inner `callClaude` call. Failed `callClaude` calls do NOT write log rows (matches the existing `trackSpendAndAlert` posture). Caller sweep covers 10 edge functions: vs-parse-brief, vs-compile-summaries (fill + overview), vs-generate-brief-overview, vs-research-single-venue, vs-regenerate-venue-overview, vs-research-venues (Phase A fill + hq-pool-enrich + fit-rescue + Phase B + url-fallback via `findVenueWebsite`), ts-pull-candidates, ts-evaluate-candidate, ts-final-review. Pre-create flow functions (ts-generate-scorecard, ts-refine-scorecard) and hq-generate-venue-about log with null ids since no scout/role is in scope.

Cron (`ts-cron-monthly-spend-reset`): 12-month prune of `anthropic_call_log` rows runs AFTER the `global_settings` reset, so a prune failure cannot block the cap-alert re-arm. Pruned count + cutoff timestamp surface in the response summary. Non-fatal: next month's run catches up.

Cap-control consolidation: new `AnthropicSpendCapCard` lifts the editable cap card from VS Settings into HQ Admin Settings (between BulkImportCard and Integrations stub) — HQ Admin Settings becomes the sole canonical cap-edit surface. TS Settings loses its cap input + cap write; VS Settings' `SpendCapCard` similarly converts to a read-only display with the cap input + Save button + R7 § F.2 placeholder banner removed. The cap value is shared across all three apps (single `global_settings.anthropic_spend_cap_monthly_usd` column); duplicate edit surfaces were a 5.15 cleanup target.
