# Edge Functions

Live in `supabase/functions/`. Each function is a directory with `index.ts`. Shared utilities live in `supabase/functions/_shared/`. Per-function settings (especially `verify_jwt`) are in `supabase/config.toml`.

Deploy via `supabase functions deploy <name>`. Don't skip JWT-verification settings: see `docs/auth-model.md` for the self-invocation pattern.

## Talent Scout

### `ts-pull-candidates(role_id, triggered_by)`
Gmail search → attachment download → PDF text extraction → link parsing → portfolio detection → Anthropic scoring. Chunked self-invoking pipeline (`BATCH_SIZE=8`) using `ts_pull_rounds.pending_candidates`. Writes one row per candidate to `ts_candidates` and one row per evaluation to `ts_evaluations`. Persists every attachment to the `candidate_attachments` bucket regardless of size (drift from source: see `docs/decisions.md`). `verify_jwt = false`.

**Referral ingestion.** When the outer Gmail `From:` is `*@mirrornyc.com` (and not `jobs@`), `parseForwardedEmail` walks the body to find the original applicant (deepest non-Mirror sender across Gmail / Apple Mail desktop + mobile / `message/rfc822` forwards); the candidate row gets the applicant's identity with `is_referral=true` + `referrer_email=<manager>`. `extractManagerNote` captures every `@mirrornyc.com` sender's commentary into `ts_candidates.internal_notes`, also folded into the FIRST evaluation via a `HIRING MANAGER NOTES:` block. History: see `docs/v1-changelog.md`.

### `ts-evaluate-candidate(candidate_id, overwrite_history?)`
Single-candidate eval/re-eval. Inserts a row into `ts_evaluations` and mirrors the latest fields onto `ts_candidates`. `overwrite_history: true` deletes prior `ts_evaluations` rows for the candidate before inserting: used by bulk re-eval (round-scoped or role-scoped) since prompt/scorecard changes invalidate prior evals. Phase 5.13.3: `_shared/buildClaudeEvalRequest.ts` `getClaudeSystemText` now injects `LOCATION_RULES` into any custom prompt that does not already contain "Location Considerations", ensuring the guardrails survive custom `evaluation_prompt` overrides. Also applies to `ts-pull-candidates` (shares the same `buildClaudeEvalRequest` helper).

### `ts-bulk-reevaluate(role_id, status_filter?)`
Re-eval the role's master pool. Chunked self-invoke. Updates `ts_roles.reeval_*` columns as it progresses; `reeval_last_progress_at` is the heartbeat read by the re-eval watchdog. `verify_jwt = false`.

PullDetail's "Re-Evaluate Pool" (round-scoped) uses a different pattern: parallel fan-out of `ts-evaluate-candidate` calls (concurrency=6) with `overwrite_history: true`, driven from the browser, not this function.

### `ts-generate-scorecard(title, job_description, hiring_priorities)`
Drafts a tiered scorecard via Claude. Called from the new-role wizard's step-3 page. Phase 3.11: each criterion now carries TWO describer fields: `full_points_rubric` (substantive 1-3 sentences for the per-candidate evaluator) and `summary` (≤ 14-word condensed recap for compact UI surfaces). One-shot, user-invoked, default `verify_jwt = true`.

### `ts-refine-scorecard(role_title, jd, hiring_priorities, criteria, ...)`: Phase 3.10
Refinement pass over a user-edited scorecard. Two call sites: the new-role wizard's step-3 page (when scorecard is "dirty since last AI pass") and the Edit Role page's RoleSettings (when scorecard has unrefined edits). Sends the current criteria + role context through Claude with `scorecardRefinementPrompt`, which preserves every concept/principle the user provided and standardizes `name` (short noun phrase), `full_points_rubric` (1-3 sentences of concrete evaluator-facing signals), and `summary` (≤ 14-word recap for compact UI surfaces; Phase 3.11). Two server-side guarantees: (1) **dead-criterion drop**: entries with `weight=0` OR with both `name` and `full_points_rubric` empty/whitespace are filtered before the prompt ever runs; the response includes a `removed_count` so the UI can surface it; (2) **defense-in-depth merge**: `tier`, `weight`, `is_disqualifier`, and `is_manual` are restored from the user's filtered input regardless of model output, so the model can never silently overwrite scoring decisions. Output count = filtered input count. Frontend re-sorts each tier highest-weight first after the refine. One-shot, user-invoked, default `verify_jwt = true`.

### `ts-final-review(role_id, top_n?, triggered_by?)`
Comparative final review across the master pool. Returns `{ final_review_id }` immediately; AI work runs in the background via `EdgeRuntime.waitUntil` and streams progress through `ts_final_reviews.step_progress` (Realtime: FinalReviewLoading subscribes). Produces `final_rankings` jsonb (`[{candidate_id, final_rank, final_tier, rationale, recruiter_note, final_overview}]`) + `pool_summary` text. Min 3 candidates; HARD_CAP=50 by total_score for context-window safety. Two-attempt JSON parse retry. Hyphen-tolerant candidate_id matching. Uses `callClaude('talent_scout', ...)`. `verify_jwt = false`.

### `ts-packet-generate(pull_round_id, top_n?, include_fast_track?)`
Round-scoped candidate review packet. Renders cover (Pulled / In Pool / Auto-Rejected / Fast-Track stats) + NYC-only top-15 comparison matrix + Fast-Track / Borderline / Other Recommended writeup pages + per-candidate title + email + attachment merge. Uploads merged PDF to the `packets` Storage bucket, updates `ts_pull_rounds.packet_*`, emails the role's hiring manager from `jobs@mirrornyc.com`. Tier subtotals render an em dash (U+2014) when `score_breakdown` is empty (Phase 3.6 Q4). `verify_jwt = false`.

### `ts-final-review-packet(final_review_id, top_n?, include_fast_track?, include_all?)`
Review-scoped packet for a completed `ts_final_reviews` row. Renders cover (Pool Size / Top Recs / Strong / Backup) + Pool Summary + full Rankings table + Top / Strong / Backup writeup pages with Not Recommended footnote + per-candidate pages. Uploads to `packets`, updates `ts_final_reviews.packet_*`, emails the hiring manager. Defaults `include_all=true` from the FinalReviewDetail UI; FinalReviewDetail's "Include Fast-Track pages" toggle controls `include_fast_track`. `verify_jwt = false`.

#### Shared infrastructure: `_shared/packetRender.ts`
~50% of source's two packet generators (CloudConvert HTML→PDF / DOCX→PDF, BASE_CSS, htmlDoc, MIRROR_LOGO_SVG, candidate title + email + packet divider renderers, Storage attachment fetcher, pdf-lib upload helper, Gmail send-with-attachment helper) lives here. HQ-specific: attachment bytes come from the `candidate_attachments` Storage bucket (Phase 3.4 persists everything on initial pull): no Gmail re-fetch at packet time. Email send uses the service account's `gmail.send` scope (added to `gmailServiceAccount.ts` SCOPES list in 3.6).

External dependency: `CLOUDCONVERT_API_KEY` Supabase secret (paid service). Packet volume is low (5-20/month) so the spend is negligible.

### `ts-send-pull-notification(role_id, pull_round_id)`: Phase 3.9
Emails the role's hiring manager when a pull completes. Tallies `ts_candidates` for the round (`fast_track`, `interview`, `consider`, `reject`), composes a plain-text body via `_shared/sendEmail.ts` and sends via the service account's `gmail.send` scope. Called fire-and-forget from `ts-pull-candidates` at every `status='complete'` write (chunked finalize, dedupe-clears-pending-to-zero, and zero-results paths). Failures are logged, not surfaced: a notification outage shouldn't fail the upstream round. Standalone in 3.9; folds into `notifications-dispatch` in Phase 5 alongside in-app bell notifications. `verify_jwt = false`.

## Talent Scout cron + watchdog (Phase 3.8)

All six are scheduled via `pg_cron` in `20260508120000_phase_3_8_cron_extensions_and_schedules.sql`. Each accepts an empty body and is invoked by `public.invoke_edge_function(fn_name, body)` (SECURITY DEFINER, reads `app.supabase_url` + `app.internal_api_secret` GUCs). All use `requireInternalOrUserAuth`. Cadences are in `docs/cron-jobs.md`.

### `ts-cron-scheduled-pulls`
Walks every `ts_roles.status='open'` AND `auto_pull_schedule != 'off'` row, computes hours-since-last-pull from `max(ts_pull_rounds.started_at)`, and self-invokes `ts-pull-candidates` for any role past its interval. Skips roles with an in-flight `running` round. Per-role failure logs but doesn't abort the loop.

### `ts-cron-pull-watchdog`
5-minute stall threshold (Phase 3.11.1, was 60). Reads `ts_pull_rounds` rows with `status='running'` AND `updated_at < now()-5m`, updates to `status='failed'` with a CAS guard (`.eq('status','running')`) so a late completion isn't overwritten. The pipeline's per-candidate row update means `updated_at` = "last candidate completed at"; a single candidate hanging >5 min is always a stall regardless of total pool size. Cron cadence bumped from every 5 min to every 2 min so detection lands within 5-7 min of stall onset.

### `ts-cron-reeval-watchdog`
30-minute stall threshold. Reads `ts_roles` rows with `reeval_status='running'` AND `reeval_last_progress_at < now()-30m`, updates to `reeval_status='failed'`.

### `ts-cron-final-review-watchdog`
20-minute stall threshold. Reads `ts_final_reviews` rows with `status='generating'` AND `generated_at < now()-20m`, updates to `status='failed'` with an `error_message` explaining the watchdog flag.

### `ts-cron-storage-cleanup`
Three-pass cleanup per `docs/cron-jobs.md`. Uses `STORAGE_BUCKET` from `_shared/attachmentStorage.ts`. Storage `.remove()` errors log and continue: a failed Storage delete leaves an orphan file but never blocks the row delete. Cron-only; no UI trigger.

### `ts-cron-monthly-spend-reset`
Resets `global_settings.anthropic_spend_current_month_usd=0` and `cap_alert_sent_this_month=false`. Logs the previous spend value for audit. **Phase 5.15:** after the global_settings reset, prunes `public.anthropic_call_log` rows older than 12 months (`lt("created_at", now - 365d)`). Prune runs AFTER the reset so a prune failure cannot block the cap-alert re-arm; prune errors are warn-logged and non-fatal (next month's run catches up). Response summary now includes `call_log_pruned_count` + `call_log_prune_cutoff`.

## Venue Scout: Phase 4

- `vs-parse-brief({ scout_id, storage_path })`: downloads a brief PDF from the `briefs` bucket and parses it via `callClaude('venue_scout', ...)` with a forced `submit_brief` tool call, returning `{ parsed_fields }` for the Brief page to merge into form state. The tool schema + sanitizers return the venue-side / event-detail fields the 3-step intake collects (`install_dates`, `strike_dates`, `activations_count`, `objectives`, `target_audience`, `vibe_aesthetic`, `target_neighborhoods`, `venue_types`, `ideal_features`); producer judgment calls (priority toggles, strict-neighborhood flag, sq-ft sliders) are intentionally NOT parsed. Tag-array fields run through a shared `sanitizeTagArray(raw, maxLen)` pass (split / drop-overlong / dedupe), `venue_types` is filtered against the DB-driven canonical set (live list rides in the per-call user message), and a re-parse guard drops `client_name` / `event_name` when the scout column is already non-empty (producer is source of truth). City names canonicalize through `resolveOrCreateCity` (alias-first ladder over `city_aliases` then `cities`, state-suffix strip, auto-INSERT on miss); alias-first ordering is load-bearing (a polluting cities row must not short-circuit canonicalization). Synchronous end-to-end (no kickoff RPC, no `EdgeRuntime.waitUntil`); user-invoked, `verify_jwt = true`. History + per-sub-phase deltas: see `docs/v1-changelog.md`; rationale: see `docs/decisions.md`.
- `vs-generate-brief-overview({ scout_id })`: generates the Event Overview paragraph for a Venue Scout brief. Reads the scout's brief fields (named columns + `brief_data` keys), calls `callClaude('venue_scout', ...)` for a 3-5 sentence overview in Mirror's voice (no custom tool, plain-text response; `SYSTEM_PROMPT` lives in the function source), then **atomically** writes `event_overview` + `brief_data.overview_source_hash` back to the scout in one UPDATE and returns `{ event_overview, overview_source_hash }`. Primary trigger is the Submit Brief click in `BriefVenue.tsx`, which fires only when the overview is empty, has no stored hash, or the stored hash no longer matches one freshly computed from the brief fields; `BriefReport.tsx` empty-state Generate + Regenerate re-invoke unconditionally. `computeOverviewSourceHash` (a 16-char SHA-256 prefix over the overview-driving fields, arrays sorted, empty-to-null normalized) is computed identically client-side in `src/lib/venue-scout/briefForm.ts` and server-side here; the two MUST stay in lockstep (same field set, canonical form, object key order). On Claude failure or empty response, falls back to a deterministic stub and persists it with the hash so the producer always lands on a non-empty editable overview. User-invoked synchronous, `verify_jwt = true` (explicit `config.toml` entry); no self-invoke. History + prompt-rewrite deltas: see `docs/v1-changelog.md`.
- `vs-parse-sheet({ scout_id, storage_path })`: parses an uploaded venue sourcing sheet (XLSX / CSV / PDF) from the `sourcing_sheets` bucket. XLSX / CSV parsed via `npm:xlsx@0.18.5` + a `pick()` fuzzy header matcher (covers `website_url` + `capacity` + keyword variants like `borough` / `pax` / `square footage`); PDF parsing is intentionally naive (returns 0 venues; frontend routes to empty-sheet error). INSERTs matched rows into `vs_candidate_venues` with `source: "sheet"` (website_url passes through `sanitizeWebsiteUrl`; producer input trusted, no HEAD check), updates `vs_scouts.sheet_storage_path`, and returns `{ count }` in sub-second time. AI enrichment lives downstream in `vs-research-venues` Phase A, not here. User-invoked synchronous, `verify_jwt = true`. History: see `docs/v1-changelog.md`.
- `vs-research-venues({ scout_id })`: AI venue work in two parallel phases inside one `EdgeRuntime.waitUntil`; returns `{ ok, scout_id }` immediately. User-invoked, `verify_jwt = true`. The Researching page Realtime-subscribes to `vs_scouts` for completion / failure. Kickoff is an atomic SECURITY DEFINER RPC `vs_research_try_acquire_kickoff({ target_scout_id, grace_seconds })` (`pg_try_advisory_xact_lock` + same-transaction read-and-write of `brief_data.research_started_at`); `false` -> 200 `{ ok, skipped: "in_flight" }`, RPC error -> 500. Idempotent on refresh via that timestamp under a 420s grace window (`IN_FLIGHT_GRACE_MS = 420_000`; 360s timeout + 60s buffer). Model `claude-sonnet-4-6`; pause_turn continuation handled in the `callClaude` wrapper.
  - **HQ Venues pool + fit filter:** `loadHqVenuesIntoPool` queries `venues WHERE city ilike scout.city`, filters out already-linked rows, scores each via `scoreVenueFitForBrief` (points table: venue_type / neighborhood / ideal_features / sq_ft / capacity, max 100) with physical-bound vetoes + aggregate-gap veto, keeps `>= 40` (top 10 ceiling per run), and INSERTs each as `source='hq_pool'` with `linked_venue_id` + `venue_overview` pre-filled from `venues.about_venue`. Idempotent across re-runs (already-linked rows skip; newly-qualifying rows land additively; a producer-edited row that cleared once survives). Skip when `scout.city` is empty. Best-effort: any failure logs + continues without the pool. Helpers are pure (caller owns telemetry).
  - **about_venue fit rescue (second-chance lane):** before Phase B, `rescueHqPoolFitVetoedVenues` sends soft-vetoed (`wrong_type` / `aggregate_gaps`) + below-threshold city-pool venues (cap 20, highest `about_venue.length` first; excludes hard physical-disqualifier vetoes, already-linked rows, and rows with no `about_venue`) to Claude via a forced `submit_fit_evaluation` tool (`{ verdicts: [{ venue_id, keep, reason }] }`) and INSERTs each keep verdict as `source='hq_pool'`. Best-effort: any failure skips rescue, deterministic pool stands, research never blocks.
  - **Phase A (sheet enrichment):** per `source='sheet'` row, `callClaude('venue_scout', { tool: fill_venue, tool_choice: auto, web_search_20250305 max_uses=2 })`; per-row patch guards never overwrite producer values; URL fallback via `findBestSearchResultUrl`. Per-row failures tolerated (`vs-compile-summaries` Pass 1 backstops).
  - **Phase B (new-venue research):** `callClaude('venue_scout', { tool: submit_research, tool_choice: auto, web_search_20250305 max_uses=4 })`. The userMsg injects a VENUE REQUIREMENTS block from the intake's `brief_data` keys (SYSTEM stays frozen per `feedback_tool_choice_collapse`). After sanitizing, **cross-rail dedupe** runs `findVenueDedupeMatch(candidate, hqPool)` from `_shared/venueDedupe.ts` against the full `city ilike scout.city` HQ pool (points ladder; address-differ veto fires); on a match to an un-seeded HQ id it does **seed-then-drop** (INSERTs that HQ venue as `source='hq_pool'` merging Claude's enrichment where HQ is null, then drops the research row); seeded-set updated in-loop. Then HEAD-validates URLs (`findVenueWebsite` per-venue Claude fallback for nulls) and INSERTs survivors.
  - **research-time hq_pool enrichment:** after `await phaseAPromise` and before the final UPDATE, `enrichHqPoolForResearch` fires the Pass 1 `fill_venue` shape on hq_pool rows still missing recs / cons / rank (independent `ENRICH_TIMEOUT_MS = 60_000` cap; never `writeFailure`). Pass 1 in `vs-compile-summaries` stays as the defense-in-depth backfill.
  - **photo pre-population (Phase 5.14):** all three hq_pool INSERT paths capture the new candidate id and `seedHqPhotosToVs` copies HQ photos from `venue_photos` into `vs_venue_photos`; fires only on fresh INSERTs; best-effort.
  - **Final write:** `derived_columns` + `current_step='sourcing_report'` + `status='in_progress'` + clears `pipeline_error`. On any failure (Claude error, no tool output, empty result, insert error, timeout), `writeFailure` sets `status='failed'` + `pipeline_error` under `.eq("current_step","researching")` CAS so a failure can't overwrite a concurrent success. `fit_meta` jsonb persistence + FitMetaIndicator deferred for v1.
  History + per-sub-phase deltas: see `docs/v1-changelog.md`. Race-fix, fit-scoring, and rescue rationale: see `docs/decisions.md`.
- `vs-compile-summaries({ scout_id })`: two-pass venue compilation via `callClaude('venue_scout', ...)`. Returns `{ ok, scout_id }` immediately; AI work runs in `EdgeRuntime.waitUntil`. User-invoked, `verify_jwt = true`. The Compiling page Realtime-subscribes to `vs_scouts`. Sequential per-venue loop; model `claude-sonnet-4-6`. Idempotency via `brief_data.compile_started_at` (300s grace; 240s timeout + 60s buffer).
  - **Pass 1 = `fill_venue` tool** (`tool_choice: auto` + `web_search_20250305 max_uses=2`): backfills any pitched venue missing fields (`venue_type` / `website_url` / `address` / `neighborhood` / `size_sq_ft` / `capacity` / `key_features` / `derived_attrs` / `recommendations` / `considerations` / `rank`), with `canonicalizeType` / `validateWebsiteUrl` / `stripPlaceholders` / `sanitizeTagShape` + URL fallback applied post-emission. `FILL_TOOL` + `FILL_SYSTEM` + `buildFillUserMsg` live in `_shared/venueFill.ts` (shared with `vs-research-venues` Phase A). `needsFill` fires for `source` in `{manual, sheet}` (recs / cons / rank / derived_attrs) and `hq_pool` (recs / cons / rank only; `derived_attrs` write gated `source !== "hq_pool"` since FILL_TOOL lacks the scout's `derived_columns` IDs); the hq_pool branch also gates on `consArr.length === 0` so seed-then-drop rows don't re-fire. `'research'` rows are already populated by Phase B. For hq_pool rows the callsite passes `aboutVenue = venue_overview` so Claude focuses on per-brief judgment (FILL_SYSTEM untouched per `feedback_tool_choice_collapse`).

The HQ Venues match-or-insert lives in `vs-generate-deck`, not here, so DeckPrep edits flow through.

- `vs-delete-scout({ scout_id })`: synchronous scout delete with storage-file cleanup (FK CASCADE handles DB rows but not Storage buckets). Sole callsite is `ScoutIndex.tsx confirmDelete`. Order is load-bearing: (1) enumerate paths (`vs_scouts.brief_data.uploaded_files` + `sheet_storage_path` + `vs_venue_photos.storage_path`); (2) DELETE `vs_scouts WHERE id = $1` (cascades to `vs_sourcing_rounds` + `vs_candidate_venues` + `vs_venue_photos`); (3) per-bucket `storage.remove(...)` against `briefs` / `sourcing_sheets` / `vs_venue_photos` in 100-path chunks. Cleanup-first ordering was rejected (a cleanup failure before the DELETE would leave a half-deleted scout). `.remove()` errors collect into `storage_errors` but never throw. Drive deck files + HQ `venues` rows (linked via `ON DELETE SET NULL`) are NOT touched by design. Auth posture: `verify_jwt = true`, service-role client, **no extra in-function authorization gate** (counter-example to `vs-research-single-venue`'s gate posture: `scout_id`-only surface with nothing to cross-poison, and open-authenticated `vs_*` RLS already lets any user delete any scout); see `docs/auth-model.md` § Function-level authorization gates. No kickoff RPC, no `EdgeRuntime.waitUntil`, no Realtime, no activity log. History: see `docs/v1-changelog.md`.

- `vs-regenerate-venue-overview({ venue_id, scout_id, notes })`: single-venue regenerate of `vs_candidate_venues.venue_overview` from the Review surface's per-card Regenerate Overview button. User-invoked synchronous; `verify_jwt = true` (default; no config.toml entry). **Authorization gates are load-bearing** (open-authenticated `vs_*` RLS leaves function-level checks as the only cross-scout poisoning defense): two gates before any Claude call, both 404 on miss (probe defense): (1) `venue.scout_id === scout_id`; (2) `scout.current_step === 'deck_prep'`. ### `bulk-import(entity_type, mode, rows, queued_refs, column_set, session_id?)`: Phase 5.9.1 (project handler 5.9.2, vendor handler 5.9.3)
Shared cross-cutting bulk-import primitive. User-invoked synchronous from `BulkImportPage`'s commit step; no self-invoke. `verify_jwt = true` (explicit `config.toml` block, matches the `vs-parse-brief` / `vs-parse-sheet` sibling pattern).

**Auth:** the gateway verifies the JWT; the function then re-checks `permission_role = 'admin'` server-side via a user-JWT client (`auth.getUser()` + a `users` SELECT). Defense in depth on top of the `AdminRoute` gate. Returns 401 for an invalid user, 403 for a non-admin (`{ ok: false, error: "Forbidden: admin only" }`).

**Signature:** `POST { entity_type: 'project'|'vendor'|'venue', mode: 'preview'|'commit'|'undo', rows, queued_refs, column_set, session_id?, dry_run? }` -> success `{ ok: true, mode, session_id?, parsed_count, created_ids?, created_refs? }` (preview/commit) or `{ ok: true, mode:'undo', dry_run, counts }` (undo); failure `{ ok: false, error, validation_errors?, unresolved_refs? }`.

**Undo mode (Phase 5.9.6):** reverts a committed import. The admin re-check runs first (same block as preview/commit), then the function requires `session_id` (else 400) and calls the SECURITY DEFINER `bulk_import_undo` RPC via the service-role client with `{ p_session_id: session_id, p_actor_id: userId, p_dry_run: dry_run ?? false }`. `entity_type` is NOT required for undo (the RPC reads it from the session row). On `dry_run: true` the RPC returns `{ ok, dry_run:true, counts }` without writing (drives the cascade-warning dialog); on `dry_run: false` it performs the destructive delete + hard-deletes the session row in one transaction and returns `{ ok, dry_run:false, counts }`. The function spreads the RPC JSON and adds `mode`. On RPC error it returns `{ ok: false, error }` with a 400 (no `failed_rollback` row — that is commit-only). See `docs/schema.md` § `bulk_import_undo` for the gates (admin, committed, 7-day window, non-empty trail) and the counts shape.

**Entity-handler registry:** a module-level `handlers: Record<string, EntityHandler>` maps each `entity_type` to a `{ validate, commit }` pair. `commit` returns `{ created_ids, created_refs, session_id? }`. Registered: `project`, `vendor`, `venue`. (The `_smoke` no-op round-trip prover handler + route was removed in 5.9.5 once all three real importers were wired.) Any unregistered `entity_type` returns `{ ok: false, error: "Entity handler not registered for '<entity>'" }` cleanly.

**RPC-owned session writes (5.9.2 / 5.9.3 / 5.9.4):** the `project`, `vendor`, and `venue` handlers' `commit()` are thin wrappers over the SECURITY DEFINER RPCs `bulk_import_commit_projects` / `bulk_import_commit_vendors` / `bulk_import_commit_venues`, which own the *entire* atomic write INCLUDING the `bulk_import_sessions` audit row + the session `activity_log` row (a PostgREST chain can't roll those back together; memory `feedback_postgrest_no_multi_statement_tx`). The handler returns the RPC's `session_id`. **When `commit()` returns a `session_id`, the edge function MUST skip its own session + activity_log inserts** to avoid a duplicate session row (the legacy non-RPC inline-write path remains only as the fallback for any future handler that returns no `session_id`). The handler passes `{ actor_id: userId, column_set, rows, queued_refs }` as the RPC payload.

**Transaction model (commit):** normalize spreadsheet-friendly values first (bare domains -> `https://...`, comma/currency-formatted numeric fields -> bare numeric strings), then validate the payload via the handler; on `preview` return the parsed count without writing. On `commit`: for RPC-backed handlers (project, vendor, venue) the RPC is the transaction (see `docs/schema.md` § `bulk_import_commit_projects` / `bulk_import_commit_vendors` / `bulk_import_commit_venues`). For any future non-RPC handler, the service-role client would write one `bulk_import_sessions` row (`status='committed'`) + one `activity_log` row (no such handler is registered today). On validation failure the 400 response's `error` includes the first row-level failures in addition to `validation_errors`; on any handler failure, a separate `bulk_import_sessions` row with `status='failed_rollback'` is written for incident traceability (outside the rolled-back work, since the RPC's own transaction has already rolled back).

**Server-side validators (`_validators/`):** the project handler's `validate` runs `_validators/project.ts` — a Deno mirror of the client-side rules (name required, single resolvable `account_lead`, valid status, ISO dates, numeric budget). The vendor handler runs `_validators/vendor.ts` (name required, `preferred` is `"true"`/`"false"`, valid email shape, parseable `website_url`, a `subcategory` requires a `category` on the same row). Last-line defense before the RPC; most validation already happens client-side in MapStep / DedupeStep / ImportGrid.

**Anthropic calls:** none. Bulk import is pure-CSV in, pure-DB out.

The HQ-side sheet parser (`_shared/hqSheetParser.ts`, fresh build) is deliberately separate from `vs-parse-sheet` so HQ-side changes can't regress the venue-scout sourcing-sheet path. The browser parses the file via the `xlsx` browser build (`src/lib/hq/bulkImport/parser.ts`) and posts already-parsed rows; per-entity re-validation runs in `_validators/`.
